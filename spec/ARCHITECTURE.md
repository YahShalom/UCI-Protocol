# UCI Technical Architecture Specification

## 1. Architectural Overview

The Universal Capability Interface (UCI) is a protocol that provides a governance and evidence control plane for heterogeneous AI agent systems. It is not an agent runtime or a tool connectivity protocol (like MCP). Instead, UCI is an architectural layer that sits above these components to enforce policy, establish trust, and generate verifiable audit trails for all agent actions.

UCI's core architectural principle is the separation of execution governance from execution implementation. It standardizes the *decision-making and auditing process* surrounding an action, allowing the action itself to be handled by external, specialized systems. This enables UCI to act as a universal control plane for a diverse and evolving agent ecosystem.

## 2. UCI Kernel

The UCI Kernel is the core logical runtime that implements the protocol. It is responsible for processing requests and managing the execution lifecycle. A Kernel can be deployed as a standalone node, a gateway, or an embedded library. Its primary responsibilities are:

*   **Intent Validation:** Parsing and validating incoming `IntentRequest` objects to ensure they are well-formed and conform to the UCI specification.
*   **Policy Enforcement:** Evaluating the `IntentRequest` and its associated `PolicyEnvelope` against the Kernel's internal policies and trust stores to decide whether the requested action is permitted.
*   **Capability Resolution:** Resolving a validated intent into a concrete `ExecutionPlan`. This involves identifying the required capabilities and structuring them into a sequence of executable steps.
*   **Evidence Generation:** Orchestrating the execution of the plan and generating a cryptographically signed `EvidenceChain` that serves as an immutable record of the entire process.

## 3. Core Protocol Objects

The UCI protocol is defined by a set of core data structures that model the execution lifecycle:

*   **`IntentRequest`**: The entry point into the UCI system. It is a declarative, structured object that captures an actor's goal, the target capability, input parameters, and an optional `PolicyEnvelope` defining execution constraints.
*   **`ExecutionPlan`**: A machine-readable sequence of steps generated by the Kernel after resolving an `IntentRequest`. It describes the exact capabilities to be executed, in what order, and with what inputs.
*   **`CapabilityDescriptor`**: A manifest file describing a specific capability. It includes metadata, input/output schemas, and, crucially, can be cryptographically signed by its publisher. The Kernel uses this signature to verify the capability's authenticity and integrity against a trust store.
*   **`EvidenceChain`**: The verifiable output of a UCI execution. It is an immutable, signed data structure containing all `EvidenceStep`s generated during the process, from intent validation to the final outcome.
*   **`Outcome`**: A final object that summarizes the result of the execution, indicating success, failure, or denial, and containing any resulting data or error messages.

## 4. Control Plane vs. Data Plane

UCI explicitly separates the control plane from the data plane.

*   **Control Plane (UCI):** The UCI Kernel operates exclusively in the control plane. It is concerned with governance, policy, trust, and auditing. It decides *if* an action should proceed, *what* steps are involved, and *records proof* of the execution.
*   **Data Plane (External Systems):** The actual execution of a capability—calling a tool, querying a database, interacting with an API—happens in the data plane. The UCI Kernel directs the data plane but does not implement its functionality. For example, a UCI Kernel might approve a plan to call a tool via the MCP protocol; UCI handles the approval, while MCP and the tool itself constitute the data plane.

This separation allows UCI to govern any agent system without needing to understand the implementation details of its tools.

## 5. Execution Flow

The UCI execution lifecycle follows a well-defined sequence managed by the Kernel:

1.  **`IntentRequest` Submission:** An actor (user or system) submits a signed `IntentRequest` to the Kernel's `execute` endpoint.
2.  **Policy Evaluation:** The Kernel validates the request and evaluates it against all applicable policies. This includes policies configured on the node and any `PolicyEnvelope` attached to the request. If the request is denied, the process terminates here, and an `Outcome` of "DENIED" is recorded.
3.  **`ExecutionPlan` Generation:** If the policy evaluation passes, the Kernel's Intent Resolver generates a detailed `ExecutionPlan`.
4.  **Capability Execution:** The Kernel's executor begins processing the `ExecutionPlan` step-by-step. For each step, it invokes the corresponding capability in the external data plane.
5.  **`EvidenceChain` Generation:** As each step of the plan is executed, a corresponding `EvidenceStep` is created, signed, and appended to the `EvidenceChain`. This provides a real-time, tamper-evident log.
6.  **`Outcome`:** Once the plan is complete (or if it fails), a final `Outcome` object is generated and returned to the original actor. The complete `EvidenceChain` is persisted for auditing.

## 6. Evidence DAG

An `EvidenceChain` is a container for `EvidenceStep`s, which are linked together to form a Directed Acyclic Graph (DAG). Each `EvidenceStep` contains a `parent_step_ids` field, allowing it to reference one or more preceding steps.

This DAG structure is more powerful than a simple linear chain, as it can accurately model complex execution flows, such as:
*   **Parallel Execution:** Multiple steps can be initiated from a single parent step.
*   **Joins/Merges:** A single step can depend on the successful completion of multiple parent steps.

The entire chain, including its graph structure and all constituent steps, is cryptographically signed, ensuring the integrity and order of the recorded events are verifiable.

## 7. Deterministic Replay

The `EvidenceChain` is a self-contained, deterministic record of an execution. It can be used to perform a **replay verification** to audit an action without re-executing it.

A replay consists of a third-party verifier taking the original `IntentRequest` and the final `EvidenceChain` and programmatically stepping through the evidence DAG. The verifier checks two things at each step:
1.  That all cryptographic signatures are valid.
2.  That the `output` of a parent step logically and correctly serves as the `input` for a child step.

This process cryptographically proves that the recorded `Outcome` is the legitimate result of the initial `IntentRequest` and the documented `ExecutionPlan`, confirming the record has not been tampered with.

## 8. Node Types

A UCI Kernel can be manifested in several forms to suit different integration needs:

*   **UCI Reference Node:** A full, standalone implementation of the UCI protocol, serving as a blueprint for developers and a platform for testing.
*   **Gateway / Sidecar Node:** A UCI Kernel packaged as an independent service that sits in front of (Gateway) or alongside (Sidecar) a non-compliant agent system to wrap its execution in a UCI governance layer.
*   **Embedded Runtime Node:** A UCI Kernel implemented as a library or package that is directly integrated into an agent's source code, enabling native UCI compliance.

## 9. Security Model

UCI's architecture is built on a foundation of "verify, then trust."

*   **Capability Signing:** `CapabilityDescriptor`s are signed by their publishers. The Kernel verifies this signature against a configurable trust store before allowing a capability to be used in an `ExecutionPlan`.
*   **Policy Envelopes:** `IntentRequest` objects can carry their own fine-grained policy rules. This allows for dynamic, context-specific security that complements the node's static base policies.
*   **Deterministic Verification:** The ability to independently replay and verify any `EvidenceChain` is the ultimate security backstop. It ensures that even if a live system were compromised, a tamper-evident record of all actions would persist, providing a clear and provable audit trail.

## 10. Deployment Topologies

The different node types support a range of flexible deployment topologies:

*   **Standalone Node:** A centralized UCI server that provides governance-as-a-service for multiple agents or services within a network.
*   **Gateway:** A proxy or API Gateway that enforces UCI policy for all traffic entering a VPC, Kubernetes cluster, or specific application.
*   **Sidecar:** In containerized environments like Kubernetes, a UCI Kernel can be deployed as a sidecar container within the same pod as an agent, intercepting all its network calls without requiring changes to the agent container itself.
*   **Embedded Runtime:** For highest performance and tightest integration, the UCI Kernel is included as a direct dependency (e.g., an npm package, a JAR file) in the agent's application code.
